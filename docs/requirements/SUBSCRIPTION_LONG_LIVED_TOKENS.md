# SUB-002: Long-Lived Subscription Tokens

> **Status**: Pending
> **Priority**: P0
> **Replaces**: SUB-001 (`.credentials.json` injection — removed entirely)
> **Prerequisite**: `claude setup-token` (works with Claude Pro and Max subscriptions)

---

## Problem

The current subscription system (SUB-001) copies `~/.claude/.credentials.json` into agent containers. This file contains a short-lived OAuth access token (~8-15 hours) and a single-use refresh token. The approach is fundamentally broken:

1. **Access tokens expire** after ~8-15 hours with no refresh mechanism inside the container
2. **Refresh tokens are single-use** (OAuth Refresh Token Rotation / RFC 9700) — when the local machine refreshes, the container's copy is permanently invalidated
3. **No daemon or process** inside the container performs token refresh
4. **Monitoring only checks file existence**, not token validity — expired tokens are never detected
5. **Race conditions** — multiple containers sharing the same refresh token invalidate each other

Users must manually re-register credentials every few hours, making autonomous agents unreliable.

## Solution

Replace the entire `.credentials.json` injection system with `CLAUDE_CODE_OAUTH_TOKEN` — a long-lived (~1 year) OAuth token generated by `claude setup-token`. The token is injected as an **environment variable** on the container, not a file.

### How `claude setup-token` Works

1. User runs `claude setup-token` locally (requires Pro or Max subscription)
2. Browser-based OAuth flow authenticates the user
3. A long-lived token is printed to stdout (prefix: `sk-ant-oat01-...`)
4. Token is valid for ~1 year, no refresh rotation
5. Can be revoked at https://claude.ai/settings → Claude Code → Authorization Tokens
6. Multiple tokens can be generated (one per use case)
7. Token uses the same rate limits as the subscription (shared pool)
8. Only works with the official `claude` CLI (not raw API calls) — this is fine since Trinity agents use `claude` CLI

### Key Simplification

| Aspect | Old (SUB-001) | New (SUB-002) |
|--------|---------------|---------------|
| **User registers** | Raw `.credentials.json` JSON blob | Single token string from `claude setup-token` |
| **Storage** | AES-256-GCM encrypted JSON blob in DB | AES-256-GCM encrypted token string in DB |
| **Injection method** | HTTP POST to agent's `/api/credentials/inject` writing a file | Container env var `CLAUDE_CODE_OAUTH_TOKEN` set at creation |
| **Token lifetime** | ~8-15 hours | ~1 year |
| **Token refresh** | Broken (no mechanism in container) | Not needed |
| **File management** | Write `.claude/.credentials.json` into running container | None — env var only |
| **Monitoring** | Check file presence (not validity) | Check env var presence (container inspect) |
| **Multiple agents** | Each gets a copy; refresh rotation breaks sharing | All share the same token safely — no rotation |
| **Container restart** | Must re-inject file after restart | Env var persists across restarts automatically |
| **Agent start flow** | Extra HTTP call to inject credentials post-start | No extra step — env var is part of container config |

## Specification

### 1. Registration

**Endpoint**: `POST /api/subscriptions` (modified)

**New request body**:
```json
{
  "name": "eugene-max",
  "token": "sk-ant-oat01-...",
  "subscription_type": "max",
  "rate_limit_tier": "default_claude_max_20x"
}
```

Changes from SUB-001:
- `credentials_json` field → `token` field (plain string, not JSON blob)
- Validation: must start with `sk-ant-oat01-` (OAuth access token prefix)
- Upsert by name (same as before)
- Token encrypted with AES-256-GCM before storage (same encryption service)

**MCP tool**: `register_subscription` — same change: `credentials_json` param → `token` param.

### 2. Database Schema

**Modified table: `subscription_credentials`**

| Column | Change |
|--------|--------|
| `encrypted_credentials` | Now stores encrypted token string (not JSON blob) |

No schema migration needed — the column type is TEXT and the content format changes transparently. The encryption/decryption layer handles both.

A one-time data migration clears any existing subscriptions registered under the old format (they contain expired tokens anyway and must be re-registered with `claude setup-token`).

### 3. Injection via Environment Variable

When a subscription is assigned to an agent:

1. **Container recreation** (same trigger as SUB-001): The container is recreated with updated env vars.
2. **Env var set**: `CLAUDE_CODE_OAUTH_TOKEN={decrypted_token}` is added to the container's environment.
3. **`ANTHROPIC_API_KEY` removed**: Same as SUB-001 — Claude Code prioritizes API key over OAuth token, so it must be absent.
4. On **subscription clear**: `CLAUDE_CODE_OAUTH_TOKEN` is removed, `ANTHROPIC_API_KEY` is restored if platform key is configured.

**No file injection needed.** The `inject_subscription_to_agent()` and `inject_subscription_on_start()` functions are removed entirely.

**Container env var logic** (in `helpers.py` `build_agent_env()`):

```python
if has_subscription:
    env["CLAUDE_CODE_OAUTH_TOKEN"] = decrypted_token
    # Do NOT set ANTHROPIC_API_KEY
elif use_platform_api_key:
    env["ANTHROPIC_API_KEY"] = platform_api_key
    # Do NOT set CLAUDE_CODE_OAUTH_TOKEN
```

### 4. Auth Mode Detection

**`get_agent_auth_mode()`** simplified:

| Condition | Auth Mode |
|-----------|-----------|
| `subscription_id` is set in `agent_ownership` | `"subscription"` |
| `use_platform_api_key` is True | `"api_key"` |
| Neither | `"not_configured"` |

No need to check agent's filesystem for `.credentials.json` — the env var is part of the container config, verifiable via Docker inspect.

### 5. Monitoring Changes

**Remove from monitoring service**:
- Credential file presence check (`/api/credentials/status` call for `.credentials.json`)
- Auto-remediation re-injection logic
- `credential_status` field on health check (or repurpose for env var check)
- `alert_subscription_credentials_missing` alert

**Optional: Add token expiry warning**:
- On registration, if the token's expiry can be extracted (JWT-like structure), store `expires_at` in the DB
- Warn when a token is within 30 days of expiry
- This is a nice-to-have, not required for MVP

### 10. Agent-Side Diagnostic Update

Update `_diagnose_exit_failure()` in `claude_code.py`:

```python
has_api_key = bool(os.environ.get("ANTHROPIC_API_KEY"))
has_oauth_token = bool(os.environ.get("CLAUDE_CODE_OAUTH_TOKEN"))

if not has_api_key and not has_oauth_token:
    return "No authentication configured. Set ANTHROPIC_API_KEY or assign a subscription token."
if not has_api_key and has_oauth_token:
    return "Subscription token may be expired or revoked. Generate a new one with 'claude setup-token'."
```

## User Workflow

### One-Time Setup

1. User runs `claude setup-token` on their local machine
2. Browser opens → authenticate with Claude Max/Pro account
3. Token is printed: `sk-ant-oat01-abc123...`
4. User registers in Trinity:
   - **Via MCP**: `register_subscription("my-max", "sk-ant-oat01-abc123...")`
   - **Via API**: `POST /api/subscriptions` with `{"name": "my-max", "token": "sk-ant-oat01-..."}`
5. Assign to agents: `assign_subscription("agent-name", "my-max")`

### Annual Renewal

1. Token expires after ~1 year
2. Agent executions start failing with auth errors
3. User runs `claude setup-token` again
4. Re-registers with same name (upsert): `register_subscription("my-max", "sk-ant-oat01-NEW...")`
5. Running agents with this subscription are **automatically restarted** to pick up the new token

### Token Revocation

1. Go to https://claude.ai/settings → Claude Code → Authorization Tokens
2. Revoke the specific token
3. Re-register with a new token in Trinity

## Migration from SUB-001

1. **Existing subscriptions**: All existing `subscription_credentials` rows contain expired `.credentials.json` blobs. They are non-functional already. On upgrade:
   - Log a warning for each existing subscription: "Subscription '{name}' uses legacy format. Re-register with `claude setup-token`."
   - Do NOT auto-delete — let admins re-register at their pace.
2. **Assigned agents**: Agents with `subscription_id` set will continue to have that assignment. On next restart, the container will be created without `CLAUDE_CODE_OAUTH_TOKEN` (since the old credential format is not a token). The diagnostic error will guide the user.
3. **Documentation**: Update `docs/SUBSCRIPTION_CREDENTIALS.md` with the new workflow.

## Known Limitations

1. **`/usage` command**: May not work with `setup-token` tokens due to missing `user:profile` scope (Claude Code issue #22450, reportedly fixed in recent versions)
2. **Token accumulation**: Each `claude setup-token` call generates a new token. Old tokens must be manually revoked at claude.ai/settings. No bulk revoke available.
3. **Max-as-Pro bug**: Historical issue where Max account tokens were generated with Pro-level access (#4085, reportedly fixed). Worth testing on first setup.
4. **Rate limit sharing**: Token usage counts against the subscription's rate limit, shared with Claude web and Claude Code interactive sessions.

## Testing

### Prerequisites
- Claude Max or Pro subscription
- `claude setup-token` run locally to obtain a token
- Trinity backend running

### Test Steps

#### 1. Register Subscription
**Action**: `POST /api/subscriptions` with `{"name": "test-max", "token": "sk-ant-oat01-..."}`
**Expected**: 200 response with subscription details
**Verify**: `GET /api/subscriptions` returns the subscription

#### 2. Assign to Agent
**Action**: `PUT /api/subscriptions/agents/my-agent?subscription_name=test-max`
**Expected**: Agent restarts, container has `CLAUDE_CODE_OAUTH_TOKEN` env var, no `ANTHROPIC_API_KEY`
**Verify**: `docker inspect agent-my-agent` → Env contains `CLAUDE_CODE_OAUTH_TOKEN`

#### 3. Execute Task
**Action**: Run a task on the agent
**Expected**: Task completes successfully using subscription auth
**Verify**: No auth errors in agent logs

#### 4. Token Persists Across Restart
**Action**: Stop and start the agent
**Expected**: Agent still has `CLAUDE_CODE_OAUTH_TOKEN` in env (container config)
**Verify**: Task execution still works after restart

#### 5. Clear Subscription
**Action**: `DELETE /api/subscriptions/agents/my-agent`
**Expected**: Agent restarts, `CLAUDE_CODE_OAUTH_TOKEN` removed, `ANTHROPIC_API_KEY` restored
**Verify**: `docker inspect agent-my-agent` → no `CLAUDE_CODE_OAUTH_TOKEN`, has `ANTHROPIC_API_KEY`

#### 6. Invalid Token
**Action**: Register with `{"name": "bad", "token": "not-a-real-token"}`
**Expected**: 400 error — token must start with `sk-ant-oat01-`

#### 7. Re-register (Upsert)
**Action**: Register same name with new token
**Expected**: Token updated, running agents with this subscription auto-restart
**Verify**: Agent container has new token value in env
